# 系统架构

> 基于《企业AI-Agent架构》PPT + 评审意见，持续更新。

## 设计目标

- **框架通用可复用**：框架层零业务概念，新业务场景可快速接入
- **业务可扩展**：业务定制通过插件/配置注入，不改框架代码
- **Agent 原生**：显式设计推理循环、记忆系统、规划策略

## 分层架构

```
┌─────────────────────────────────────────────────────────┐
│                      表现层                               │
│        Web | 移动端 | CLI | 飞书 | 语音端                 │
├─────────────────────────────────────────────────────────┤
│                    应用交互层                              │
│    意图理解 | 会话管理 | 上下文管理 | 身份认证              │
├─────────────────────────────────────────────────────────┤
│                     编排层（Agent 核心）                   │
│  ReAct 推理循环 | Memory 三层记忆 | Planning 策略          │
│  Tool 调用 | 多 Agent 编排 | DAG 调度                    │
├─────────────────────────────────────────────────────────┤
│                     服务协议层                             │
│  多模型适配器 | MCP | REST/gRPC | OAuth/JWT/mTLS         │
│  流式输出（SSE/WebSocket）                               │
├─────────────────────────────────────────────────────────┤
│                     基础设施层                             │
│  配置中心 | 日志 | 存储 | 缓存 | 事件总线 | 状态机 | 权限  │
└─────────────────────────────────────────────────────────┘
```

## 通用框架层 vs 业务层边界

**判断标准：一个模块是否包含任何业务概念。**

### ✅ 通用层
- 事件总线、状态管理、插件加载器
- 日志框架、配置中心
- 网络封装、文件管理、缓存系统
- ReAct 推理循环、Memory 三层架构、Planning 策略引擎

### ❌ 不属于通用层
- 任务拆解引擎（业务编排的一部分）
- 策略引擎（规则 100% 业务定义）
- 业务专用 Tool、意图库、领域词汇表

## Agent 核心机制

### ReAct 推理循环

```
Thought (推理) → Action (执行工具) → Observation (观察结果)
     ↑                                        ↓
     └────────────────────────────────────────┘
```

**终止条件：** 任务完成 / 达到最大轮次 / 主动交还人类 / 不可恢复错误

### Memory 三层记忆架构

| 层级 | 作用域 | 读取策略 | 写入策略 |
|------|--------|---------|---------|
| 工作记忆 | 单轮推理 | 每轮全量读取 | Observation 自动追加 |
| 短期记忆 | 单次会话 | 按需检索（语义 Top-K） | 会话结束时自动摘要写入 |
| 长期记忆 | 跨会话 | 短期记忆不足时触发 | 用户确认的关键信息 |

**待补充：** 遗忘机制（淘汰策略、记忆压缩、重要性评分）

### Planning 策略自动选择

| 场景 | 策略 |
|------|------|
| 单意图 + 单工具 | CoT（Chain of Thought） |
| 单意图 + 多步骤（>3步） | Plan-and-Solve |
| 多意图 + 多工具协作 | ReWOO |
| 用户指定 | 手动覆盖自动选择 |

### Tool 调用决策

```
语义匹配 → 上下文推理 → 参数填充 → 执行调用
```

- 每个 Tool 注册：名称、描述、输入 Schema、输出 Schema
- 支持工具组合（串行/并行）
- **待补充：** 并发控制、超时策略、幂等性保证

## Multi-Agent 策略

**推荐：先单 Agent + Pipeline，再按需引入 Multi-Agent。**

迁移路径：单 Agent → 增加 Supervisor → 拆分子 Agent → 引入 Agent 间通信

## 各层设计原则

### 表现层
- **目标：** 统一交互入口，屏蔽底层差异
- **原则：** 表现层只负责展示和采集，不包含业务逻辑
- 详细设计待补充

### 应用交互层
- **目标：** 用户意图理解与会话管理
- **原则：** 意图解析与业务解耦，通过配置注册意图
- 详细设计待补充

### 编排层（Agent 核心）
- **目标：** 推理、记忆、规划、工具调用的核心引擎
- **原则：** 通用框架层，零业务概念；ReAct/Memory/Planning 可直接跨业务复用
- 详细设计待补充

### 协议层
- **解决的问题：** AI 系统的核心依赖是各种外部模型和服务，它们接口各异、协议不同、部署方式不同（云端/本地）。协议层的职责是将这些异构能力统一抽象，让上层只依赖标准接口，不关心底层实现。
- **设计目标：**
  - 上层（编排层）只面向统一接口编程，切换模型零改动
  - 新增模型供应商只需实现适配器，不改框架代码
  - 支持云端模型和本地模型无缝切换
  - 流式输出、降级、限流等能力内聚在协议层
- **设计原则：**
  - 依赖倒置：上层依赖抽象接口，不依赖具体适配器实现
  - 接口稳定：接口定义后尽量避免破坏性变更，新能力通过扩展实现
  - 配置驱动：模型选择、参数、降级策略通过配置管理
- **核心模块：**
  - 模型适配器（LLM / Embedding / ASR / TTS / 多模态）→ [详细设计](design/06-协议层-模型适配器.md)
  - MCP 协议 → 待详细设计
  - 协议转换（REST/gRPC） → 待详细设计
  - 认证授权 → 待详细设计

### 基础设施层
- **目标：** 纯技术底座，零业务概念
- **原则：** 提供配置、日志、存储、缓存、事件、状态等基础能力，不包含任何业务语义
- 详细设计待补充

## 横切关注点

### 错误处理
- 模型降级：主模型 → 备用模型 → 规则兜底
- 工具降级：核心不可用 → 跳过非关键步骤 → 返回部分结果
- 外部服务降级：不可写 → 本地暂存 → 延迟重试
- 熔断机制：连续失败 N 次后暂停，半开探测恢复

### 可观测性
- 统一 Trace ID 贯穿所有层
- 告警去重合并 + 按时间升级通知

### 安全性
**待补充：** Prompt Injection 防护、工具调用权限控制、用户数据隔离
