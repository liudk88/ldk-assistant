# 开发规范

> 每次功能开发必须遵守的流程与规范。

## 核心原则

**文档优先于代码。** 开发只是工作的一部分，设计文档的规范性和可落地性是更重要的产出。代码可以重写，但架构决策和设计沉淀是长期资产。

## 设计原则

### 思考发散，实现聚焦

- **讨论阶段发散思考** — 充分考虑未来各种可能性，目标：避免日后扩展时做重大调整
- **实现阶段聚焦当下** — 只做当前确定需要的东西，避免过度设计

### 预留接口，增量扩展

- 当前确定会有的能力，直接做（如：本地模型 + 云端模型都确定需要 → 都做）
- 当前只用到其中一个，但另一个已确定未来需要 → 先做用到的那个，**但定义好统一接口**，未来扩展只需增量加代码，不改动已有代码
- 未来不确定是否需要的能力 → 只做接口预留（如定义空接口），不做实现

### 判断标准

| 场景 | 做法 | 举例 |
|------|------|------|
| 确定需要，当前就要用 | 立即实现 | 云端模型 + 本地模型适配器都做 |
| 确定需要，当前用不到 | 定义接口，不实现 | 先做 OpenAI 适配器，但 ModelProvider 接口要能支持 Ollama |
| 不确定是否需要 | 不做，不预留 | JWT 认证暂不需要，不提前实现 |

### 接口设计要求

- 接口一旦定义，尽量不要破坏性变更
- 新增能力通过实现新接口 / 扩展已有接口，不修改已有接口签名
- 遵循依赖倒置：上层依赖抽象接口，不依赖具体实现

## 功能迭代开发流程

```
1. 回顾文档 → 2. 评估可行性 → 3. 开发（或优化文档） → 4. 更新文档 → 5. 提交
```

### Step 1 — 回顾设计文档

每次开发新功能前，**必须**回顾以下文档：

| 文档 | 回顾重点 |
|------|---------|
| **01-系统架构.md** | 该功能放在哪一层？是否引入新的分层？与现有模块的关系 |
| **02-功能架构.md** | 该功能是否已在功能列表中？是否影响已有功能的边界定义 |
| **03-技术路线图.md** | 该功能的技术选型是否符合已定的技术栈？是否引入新技术 |
| **04-功能开发清单.md** | 该功能属于哪个阶段？是否提前引入了复杂度 |

### Step 2 — 评估可行性

回顾文档后，判断是否存在冲突：

**✅ 无冲突 → 直接开发**

**⚠️ 有冲突 → 分两种情况：**
- **文档需要优化** → 先更新文档，再开发。架构和设计必须走在代码前面
- **功能需要调整** → 重新评估功能范围，可能降级、拆分或延后

**冲突示例：**
- 新功能需要引入技术路线图 v2 才计划的技术 → 评估是否提前引入，还是降级实现
- 新功能模糊了通用层与业务层的边界 → 先明确边界定义
- 新功能与已有功能的职责重叠 → 先调整功能架构图

### Step 3 — 类设计（开发前）

每次功能开发前，**必须**先做类设计，产出以下内容：

- **目录结构** — 明确每个目录的职责
- **类/文件职责** — 每个类或文件做什么，一句话说清
- **方法职责** — 每个公开方法的输入、输出、作用
- **协作关系** — 各模块之间如何调用配合，完成整体功能

类设计以注释或文档形式写在代码中，保证代码自解释。

### Step 4 — 编码

- 按功能粒度开发，每个功能一个 commit 或一个 PR
- 遵循已定的技术栈，不私自引入未经讨论的技术
- 编写对应的单元测试（见「测试规范」）

### Step 5 — 更新文档

开发完成后，**必须**同步更新文档：

- **04-功能开发清单.md** — 对应条目打勾 ✅
- **02-功能架构.md** — 如果功能有变化，更新描述
- **03-技术路线图.md** — 如果实际技术选型有调整，记录变更及原因
- **01-系统架构.md** — 如果影响架构，更新架构图

### Step 6 — 提交

```
git add .
git commit -m "feat: <功能简述>"
git push
```

## 提交规范

使用 Conventional Commits：

| 前缀 | 用途 |
|------|------|
| `feat:` | 新功能 |
| `fix:` | 修复 |
| `docs:` | 文档更新 |
| `refactor:` | 重构 |
| `chore:` | 构建/工具/配置 |

## 编码规范

### KISS 原则

- 单个方法/函数**不超过 60 行**
- 单一职责：一个方法只做一件事
- 逻辑复杂时，拆分为多个小方法，每个方法可独立理解

### 测试规范

- 独立目录存放单元测试：`__tests__/`（与源码目录结构对应）
- 每个功能必须编写单元测试，覆盖核心逻辑和边界情况
- **所有单元测试通过后，才能进入下一个功能的开发**
- 测试先行优先：复杂逻辑先写测试用例，再写实现

### 项目结构示例

```
packages/core/
├── src/
│   ├── agent.ts          # Agent 核心类
│   ├── memory/
│   │   ├── index.ts      # Memory 管理器
│   │   ├── working.ts    # 工作记忆
│   │   └── long-term.ts  # 长期记忆
│   └── tool/
│       ├── index.ts      # Tool 调度器
│       └── registry.ts   # Tool 注册表
└── __tests__/
    ├── agent.test.ts
    ├── memory/
    │   ├── working.test.ts
    │   └── long-term.test.ts
    └── tool/
        ├── dispatcher.test.ts
        └── registry.test.ts
```

## 禁止事项

- ❌ 不回顾文档直接开发
- ❌ 引入未经讨论的新技术栈
- ❌ 代码提交不更新对应文档
- ❌ 跳过功能开发清单直接进入下一阶段功能
- ❌ 单个方法超过 60 行
- ❌ 不写单元测试就提交代码
- ❌ 单元测试未全部通过就开发下一个功能
